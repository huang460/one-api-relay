# 小米「百亿 Token 计划」— 项目描述

> 以下内容直接对应申请表中"使用 Agent 或 AI 驱动构建的具体成果"字段

---

我构建了一个基于 One API 的 **AI 大模型统一管理与分发平台 (One API Relay)**，作为团队 AI 应用的基础设施层。

**1. 核心痛点**

在 AI Agent 开发过程中，团队面临以下关键问题：
- **多模型碎片化**：团队同时使用 OpenAI、Claude、DeepSeek、智谱、通义千问等多个模型，各厂商 API 格式和鉴权方式不同，Agent 需要为每个模型单独适配
- **Key 管理混乱**：多个 API Key 分散在开发者手中，缺乏统一管控，存在 Key 泄露和滥用风险
- **Token 成本不可控**：缺乏用量统计和预算管控，月底对账困难
- **国内访问不稳定**：直连海外 API 延迟高、时常中断，影响 Agent 稳定性

**2. 核心逻辑流**

平台的完整请求链路由以下模块协同完成：

```
用户/Agent 发起请求
       │
       ▼
  ┌─ 统一网关层 ──────────────────┐
  │  兼容 OpenAI SDK 格式          │
  │  用户鉴权 + 配额检查             │
  └──────────┬────────────────────┘
             │
             ▼
  ┌─ 智能调度层 ──────────────────┐
  │  Key 池轮换（多 Key 负载均衡）   │
  │  故障自动切换（降级到备用厂商）    │
  │  模型路由（按策略分发到最优模型）   │
  └──────────┬────────────────────┘
             │
             ▼
  ┌─ 多厂商适配层 ────────────────┐
  │  OpenAI / Claude / DeepSeek   │
  │  文心 / 通义 / 混元 / ChatGLM  │
  │  Gemini / Moonshot / xAI...   │
  └──────────┬────────────────────┘
             │
             ▼
  ┌─ 监控统计层 ──────────────────┐
  │  Token 用量实时统计             │
  │  请求延迟与错误率监控            │
  │  用户级别配额管理               │
  └───────────────────────────────┘
```

**长链推理支持**：平台的智能调度层内置了 Key 健康度评分算法，当某个 Key 频繁触发 rate limit 或返回错误时，自动降低其权重并将流量切换到健康 Key，实现无需人工干预的闭环运维。

**多 Agent 协作场景**：在典型使用场景中，多个 AI Agent 同时通过平台调用不同模型——例如 RAG Agent 调用文本嵌入模型，对话 Agent 调用推理模型，代码 Agent 调用代码生成模型。平台根据 Agent 的类型和优先级，动态分配资源和配额，确保高优先级 Agent 获得稳定的服务质量。

**3. 落地效果**

目前已将平台部署到本地环境并通过 ngrok 公网暴露，实现了：
- 30+ AI 厂商的统一接入，Agent 无需关心底层模型切换
- 多 Key 自动轮换，单 Key 故障不影响整体服务
- 完整的 Token 用量统计，每笔请求均可追溯
- Web 管理面板可视化运维

该项目已开源至 GitHub（https://github.com/huang460/one-api-relay），为更多 AI 开发者提供开箱即用的 API 管理基础设施。
